Fundamentos

1 ¿Qué es Cypress?

Cypress es un framework de testing end-to-end (E2E) para aplicaciones web, pensado desde cero para JavaScript/TypeScript. A diferencia de herramientas como Selenium, Cypress no controla el navegador "desde afuera" mediante un protocolo remoto: se ejecuta dentro del mismo navegador, en el mismo ciclo de eventos que tu aplicación. Esto le da acceso directo al DOM, a la red, al window y al almacenamiento, lo que se traduce en tests más rápidos, más estables y mucho más fáciles de depurar.

¿Por qué se volvió tan popular?

⏱Auto-espera (auto-wait) sin sleeps manuales
↺Time-travel: ves cada paso del test
TSSoporte nativo de TypeScript
🌐Mock de red real con cy.intercept
  • Auto-waiting / retry-ability: antes de fallar, Cypress reintenta automáticamente comandos y aserciones durante un tiempo configurable (por defecto 4s), así que no necesitás sleep() ni wait(2000) a mano.
  • Time travel: el Test Runner guarda una "foto" del DOM en cada comando, podés pasar el mouse por cada paso y ver exactamente qué pasó.
  • Debugging real: como corre en el navegador, podés abrir las DevTools de Chrome igual que con cualquier página y poner debugger.
  • Control de red: con cy.intercept() podés espiar, modificar o mockear cualquier request sin tocar el backend.
  • Todo en un solo paquete: test runner, librería de aserciones (Chai), mocking y documentación, sin tener que combinar mil librerías distintas.

Para testing E2E de una SPA o sitio web típico, Cypress es una de las mejores opciones del mercado.

2 Repaso rápido de JavaScript y TypeScript

Antes de meternos en Cypress, repasemos lo mínimo de JavaScript y TypeScript que vas a usar todo el tiempo al escribir tests. Si ya te sentís cómodo con esto, igual valen unos minutos de repaso porque son los bloques con los que se arma cada test.

Variables: const y let

En TypeScript moderno casi no se usa var. Usá const por defecto (no se puede reasignar) y let sólo cuando sepas que el valor va a cambiar.

TS
const usuario = "ana@mail.com";   // no se puede reasignar
let intentos = 0;                  // si se puede reasignar
intentos = intentos + 1;

Tipos básicos de TypeScript

TypeScript agrega tipos estáticos sobre JavaScript: el editor te avisa de errores antes de correr el código. Esto es oro en testing, porque evita typos en nombres de propiedades de objetos grandes (fixtures, respuestas de API, etc.).

TS
let email: string = "ana@mail.com";
let edad: number = 28;
let activo: boolean = true;
let tags: string[] = ["admin", "qa"];
let coords: [number, number] = [10, 20]; // tupla

// "object shape" con interface
interface Usuario {
  id: number;
  nombre: string;
  email: string;
  esAdmin?: boolean; // el "?" significa opcional
}

const ana: Usuario = { id: 1, nombre: "Ana", email: "ana@mail.com" };
TIP any vs. tipado real

Vas a sentir la tentación de tipar todo como any cuando TypeScript se queje. Resistila: any apaga el chequeo de tipos justo donde más lo necesitás. Preferí unknown si realmente no sabés el tipo, o definí la interface correspondiente.

Funciones flecha (arrow functions)

Cypress usa funciones flecha (() => { }) en casi todos lados: en cada it(), describe(), callbacks de .then(), etc.

TS
// funcion tradicional
function sumar(a: number, b: number): number {
  return a + b;
}

// arrow function equivalente
const sumar2 = (a: number, b: number): number => a + b;

// asi se ven en Cypress
describe("Carrito de compras", () => {
  it("agrega un producto", () => {
    // aca va el test
  });
});

Destructuring y template strings

TS
const usuario = { nombre: "Ana", email: "ana@mail.com" };
const { nombre, email } = usuario;          // destructuring

const saludo = `Hola, ${nombre}!`;          // template string
console.log(saludo); // "Hola, Ana!"

Promesas y async/await

En JavaScript "normal", el código asincrónico (que tarda, como un fetch) se maneja con Promise y async/await:

TS
async function traerUsuario() {
  const resp = await fetch("/api/usuario/1");
  const data = await resp.json();
  return data;
}
ATENCIÓN Cypress NO usa async/await

Esto es clave y confunde a mucha gente que recién empieza: dentro de un test de Cypress casi nunca vas a escribir await. Cypress arma internamente una cola de comandos y los encadena con .then(). Vas a ver esto en detalle en la sección 6 (Anatomía de un test), pero quedate con la idea: se programa distinto a como esperarías con promesas tradicionales.

3 Instalación del entorno completo

Vamos a instalar todo desde cero: Node.js, un proyecto nuevo, Cypress, y la configuración de TypeScript. Seguí los pasos en orden.

  1. Instalar Node.js

    Cypress necesita Node.js (incluye npm). Descargalo desde nodejs.org (versión LTS) o instalalo con un gestor de versiones como nvm. Verificá la instalación:

    BASH
    node --version   # ej: v20.11.0
npm --version    # ej: 10.2.4
  2. Crear la carpeta del proyecto
    BASH
    mkdir mi-proyecto-cypress
cd mi-proyecto-cypress
npm init -y

    npm init -y crea un package.json con valores por defecto, donde se van a guardar las dependencias del proyecto.

  3. Instalar Cypress y TypeScript
    BASH
    npm install --save-dev cypress typescript

    Cypress trae soporte de TypeScript integrado: no hace falta ningún transpilador adicional, sólo un tsconfig.json bien configurado (paso 6).

  4. Abrir Cypress por primera vez
    BASH
    npx cypress open

    Esto abre el Test Runner gráfico. Te va a preguntar qué tipo de testing querés configurar:

    • E2E Testing — para probar la app completa como un usuario real (es el foco de esta guía).
    • Component Testing — para montar y probar componentes de UI de forma aislada.

    Elegí E2E Testing y seguí el asistente: Cypress detecta el navegador instalado y crea automáticamente la estructura de carpetas.

  6. Configurar TypeScript dentro de Cypress

    Cypress genera cypress.config.ts automáticamente. Necesitás además un tsconfig.json dentro de la carpeta cypress/ (separado del tsconfig de tu app, si lo tenés):

    cypress/tsconfig.json
    {
  "compilerOptions": {
    "target": "es5",
    "lib": ["es5", "dom"],
    "types": ["cypress", "node"],

    "strict": true,

    Si tu proyecto principal (la app) ya tiene su propio tsconfig.json, lo mejor es que Cypress tenga el suyo dentro de cypress/. Así evitás que la configuración de build de tu app interfiera con la de los tests (y viceversa).

    Acá vive la configuración general: URL base, patrones de archivos de test, timeouts, variables de entorno, etc.

    Si usás VS Code, instalá la extensión oficial Cypress (de Cypress.io) para autocompletado de comandos, y asegurate de tener la extensión de ESLint activa. No es obligatorio, pero ahorra muchísimo tiempo.

4 Primer proyecto Cypress + TypeScript

Con todo instalado, vamos a escribir y correr tu primer test. La convención de Cypress es que los archivos de test terminen en .cy.ts y vivan dentro de cypress/e2e/.

Tu primer archivo de test

cypress/e2e/home.cy.ts
// cypress/e2e/home.cy.ts

describe("Pagina de inicio", () => {
  it("muestra el titulo principal", () => {
    cy.visit("/");
    cy.get("h1").should("be.visible");
    cy.get("h1").should("contain.text", "Bienvenido");
  });
});

Dos formas de correr los tests

ComandoModoCuándo usarlo
npx cypress openInteractivo, con interfaz gráficaMientras escribís y debuggeás tests
npx cypress runHeadless, por consolaEn CI/CD o para correr todo rápido

Qué vas a ver en el Test Runner

  • A la izquierda, la lista de comandos ejecutados (visit, get, should...), en orden.
  • A la derecha, tu aplicación corriendo de verdad, dentro de un iframe.
  • Si pasás el mouse sobre cualquier comando de la izquierda, el panel de la derecha "viaja en el tiempo" y te muestra cómo estaba el DOM en ese momento exacto (esto es el famoso time-travel).
  • Si un comando falla, Cypress reintenta automáticamente hasta el timeout configurado antes de marcar el test como fallido, y te muestra el error con una captura de pantalla.
NOTA baseUrl: tu mejor amigo

Notá que en el ejemplo usamos cy.visit('/') en vez de la URL completa. Eso es porque definimos baseUrl en cypress.config.ts. Siempre que tu app corra en una URL fija (local o de staging), configurá baseUrl: así los tests son portables entre entornos sin tocar el código.

5 Selectores CSS: la base para encontrar elementos

Cypress encuentra elementos en la página usando selectores CSS (los mismos que usás para aplicar estilos). Si dominás CSS, ya sabés encontrar cualquier elemento con cy.get(). Repasemos los selectores de la forma en que más se usan en testing.

INFO El selector va siempre dentro de cy.get()

Todo lo que veas en esta sección se usa así: cy.get("SELECTOR"). Por ejemplo, cy.get(".btn-primary") selecciona todos los elementos con clase btn-primary, igual que en una hoja de estilos.

5. Especificidad: quién gana cuando hay conflicto

La especificidad determina qué regla CSS "gana" si varias aplican al mismo elemento. Se calcula como una tupla (inline, IDs, clases/atributos/pseudo-clases, elementos). Para testing rara vez vas a tener "conflictos" como en CSS de diseño, pero entender esto te ayuda a elegir el selector más específico y estable posible.

BUENA PRÁCTICA Selectores estables vs. selectores frágiles

Un selector como div > div:nth-child(3) > span es muy específico, pero se rompe apenas alguien cambia el HTML. Un atributo dedicado como [data-cy="precio-total"] es mucho más resistente a cambios de diseño o de estructura. Volvemos sobre esto en la sección 13.

6 Anatomía de un test en Cypress

Todo test de Cypress se arma con bloques de Mocha (el framework de test sobre el que corre Cypress) más Chai (la librería de aserciones). Vamos a desarmar un test completo, pieza por pieza.

Los bloques estructurales

BloquePara qué sirve
describe()Agrupa tests relacionados (una "suite")
context()Alias de describe(), útil para sub-agrupar
it()Un caso de prueba individual
before()Corre 1 sola vez, antes de todos los tests del bloque
beforeEach()Corre antes de CADA test del bloque
after()Corre 1 sola vez, al final de todos los tests
afterEach()Corre después de CADA test

Test completo y comentado

cypress/e2e/login.cy.ts
describe("Login", () => {
  beforeEach(() => {
    // se ejecuta antes de cada test: dejamos la app en un estado conocido
    cy.visit("/login");
  });

  it("permite iniciar sesion con credenciales validas", () => {
    cy.get('[data-cy="email"]').type("ana@mail.com");
    cy.get('[data-cy="password"]').type("Secreta123!");
    cy.get('[data-cy="submit"]').click();

    cy.url().should("include", "/dashboard");
    cy.get('[data-cy="bienvenida"]').should("contain.text", "Hola, Ana");
  });

  it("muestra un error con credenciales invalidas", () => {
    cy.get('[data-cy="email"]').type("ana@mail.com");
    cy.get('[data-cy="password"]').type("incorrecta");
    cy.get('[data-cy="submit"]').click();

    cy.get('[data-cy="mensaje-error"]')
      .should("be.visible")
      .and("contain.text", "Credenciales invalidas");
  });
});

El "encadenamiento" (chaining): la idea más importante de Cypress

Cuando escribís cy.get(...).click(), NO estás llamando a dos funciones que se ejecutan inmediatamente una tras otra como en JS normal. Cypress encola comandos y los va resolviendo en orden, esperando automáticamente a que cada uno esté listo.

TS
// Esto NO ejecuta get() y luego click() al instante.
// Cypress arma una "cola": primero resuelve get(), reintentando
// hasta encontrar el elemento, y RECIEN AHI ejecuta click().
cy.get('[data-cy="boton"]').click();

// Por eso esto NO funciona como esperarias en JS normal:
const boton = cy.get('[data-cy="boton"]'); // esto NO es el elemento todavia
// boton.click()  -> error conceptual, cy.get() devuelve una "Chainable", no el DOM
ATENCIÓN No guardes resultados de cy.get() en variables comunes

Como en el ejemplo de arriba: cy.get() no devuelve el elemento del DOM directamente, devuelve un objeto especial (Chainable) que Cypress resuelve más tarde. Si necesitás "guardar" algo para usarlo después, la forma correcta es con .then() o con alias (.as()), que vas a ver en la sección 7.

Usando .then() para trabajar con el valor real

TS
cy.get('[data-cy="precio"]').then(($el) => {
  const texto = $el.text();              // aca SI tenes el elemento real (jQuery)
  const precio = parseFloat(texto.replace("$", ""));
  expect(precio).to.be.greaterThan(0);
});

8 Aserciones (assertions)

Las aserciones son las que realmente "verifican" algo: sin ellas, un test sólo navega y hace clics sin comprobar nada. Cypress usa Chai (más plugins como chai-jquery) por debajo, expuesto principalmente a través de .should().

Dos estilos de aserción

EstiloSintaxisCuándo usarlo
Implícito (BDD)cy.get(...).should(...)El 95% de los casos: aprovecha el retry automático de Cypress
Explícitoexpect(valor).to.equal(...)Dentro de un .then(), cuando ya tenés el valor en la mano
TS
// Implicito: Cypress reintenta esta asercion hasta que sea verdadera (o haga timeout)
cy.get('[data-cy="titulo"]').should("be.visible");

// Explicito: una sola comprobacion puntual, sin reintentos automaticos
cy.get('[data-cy="precio"]').then(($el) => {
  const valor = parseFloat($el.text());
  expect(valor).to.be.greaterThan(0);
});
INFO ¿Por qué importa la retry-ability?

10 Comandos personalizados (custom commands)

Cuando un mismo flujo se repite en muchos tests (el típico ejemplo: el login), conviene extraerlo a un comando personalizado. Se definen en cypress/support/commands.ts y quedan disponibles como cy.miComando() en cualquier test.

Definir el comando

cypress/support/commands.ts
// cypress/support/commands.ts

Cypress.Commands.add("login", (email: string, password: string) => {
  cy.visit("/login");
  cy.get('[data-cy="email"]').type(email);
  cy.get('[data-cy="password"]').type(password);
  cy.get('[data-cy="submit"]').click();
  cy.url().should("include", "/dashboard");
});

Tiparlo para TypeScript (paso clave, no opcional)

Si no hacés esto, TypeScript va a marcar error en cy.login(...) porque no sabe que ese comando existe. Hay que extender la interface global Cypress.Chainable:

cypress/support/index.d.ts
// cypress/support/index.d.ts  (o al final de commands.ts)

declare global {
  namespace Cypress {
    interface Chainable {
      /**
       * Inicia sesion con email y password, y espera a llegar al dashboard.
       * @example cy.login("ana@mail.com", "Secreta123!")
       */
      login(email: string, password: string): Chainable<void>;
    }
  }
}

export {};

Usarlo en cualquier test

TS
describe("Dashboard", () => {
  beforeEach(() => {
    cy.login("ana@mail.com", "Secreta123!"); // una linea en vez de 5
  });

  it("muestra el nombre del usuario", () => {
    cy.get('[data-cy="bienvenida"]').should("contain.text", "Ana");
  });
});

11 Interceptación de red con cy.intercept

cy.intercept() te permite observar, modificar o reemplazar cualquier request de red que haga tu aplicación, sin tocar el backend. Es una de las funcionalidades que más distingue a Cypress de otras herramientas.

Sintaxis básica

TS
cy.intercept(metodo, url, respuesta?)

Caso 1: sólo espiar (sin modificar nada)

TS
cy.intercept("POST", "/api/login").as("loginRequest");
cy.get('[data-cy="submit"]').click();
cy.wait("@loginRequest").its("response.statusCode").should("eq", 200);

Caso 2: mockear la respuesta completa

TS
cy.intercept("GET", "/api/productos", {
  statusCode: 200,
  body: [
    { id: 1, nombre: "Zapatillas", precio: 15000 },
    { id: 2, nombre: "Remera", precio: 5000 },
  ],
}).as("getProductos");

cy.visit("/catalogo");
cy.wait("@getProductos");
cy.get('[data-cy="producto"]').should("have.length", 2);

Caso 3: mockear con una fixture (más prolijo)

TS
cy.intercept("GET", "/api/productos", { fixture: "productos.json" }).as("getProductos");

Caso 4: simular un error del servidor

TS
cy.intercept("GET", "/api/productos", {
  statusCode: 500,
  body: { error: "Error interno del servidor" },
}).as("getProductosError");

cy.wait("@getProductosError");
cy.get('[data-cy="mensaje-error"]').should("be.visible");

Caso 5: usar un wildcard / patrón de URL

TS

El Page Object Model (POM) es un patrón de diseño: en vez de escribir selectores CSS sueltos en cada test, los encapsulás dentro de una clase que representa una página (o un componente). El test queda leyendo como una descripción del flujo de negocio, no como una lista de selectores.

13 Buenas prácticas

Esta es probablemente la sección más importante para escribir tests que el equipo pueda mantener en el tiempo sin volverse loco. Son las prácticas recomendadas oficialmente por Cypress, más las que se aprenden con experiencia real en proyectos grandes.

1. Usá atributos data-cy para seleccionar elementos

Es la práctica más importante de todas. No selecciones por clases CSS, IDs pensados para estilos, ni por texto que puede cambiar con la traducción del sitio.

CSS
<!-- En tu HTML/JSX -->
<button class="btn btn-primary mt-2" data-cy="submit-button">Enviar</button>
TS
// MAL: si el equipo de diseño cambia la clase, el test se rompe
cy.get(".btn-primary").click();

// BIEN: el atributo data-cy no tiene ninguna otra funcion mas que testing
cy.get('[data-cy="submit-button"]').click();
BUENA PRÁCTICA ¿Por qué data-cy y no data-test o data-testid?

Cualquiera de los tres funciona igual de bien: lo importante es elegir uno y usarlo siempre en todo el proyecto. data-cy es la convención que sugiere la documentación oficial de Cypress.

2. Los tests deben ser independientes entre sí

Cada it() tiene que poder correr solo, en cualquier orden, sin depender de que otro test haya corrido antes.

TS
// MAL: el segundo test depende de que el primero haya creado el usuario
it("crea un usuario", () => { /* ... */ });
it("edita el usuario creado antes", () => { /* ... */ }); // fragil

// BIEN: cada test deja la app lista desde cero (beforeEach, o vía API)
beforeEach(() => {
  cy.request("POST", "/api/test/seed-usuario"); // crea datos por API, no por UI
});
it("edita un usuario", () => { /* ... */ });

3. Nunca uses esperas de tiempo fijo

TS
// MAL: tiempo arbitrario, lento si sobra, fragil si falta
cy.wait(3000);
cy.get('[data-cy="resultado"]').should("be.visible");

// BIEN: esperá una condicion real, o un request interceptado
cy.intercept("GET", "/api/resultado").as("getResultado");
cy.wait("@getResultado");
cy.get('[data-cy="resultado"]').should("be.visible"); // ademas, reintenta solo

4. Creá datos de prueba por API, no por UI, cuando sea posible

TS
// Lento: pasar por 5 pantallas de formulario solo para tener un usuario de prueba
cy.visit("/registro");
cy.get(...).type(...);
// ...

// Rapido: si solo necesitas que el usuario EXISTA para testear otra cosa
cy.request("POST", "/api/test/usuarios", { email: "ana@mail.com" });
cy.visit("/login");
TIP Reservá la UI para lo que realmente estás probando

Si tu test trata sobre el checkout, no hace falta registrar el usuario paso a paso por pantalla: creá el usuario por API y andá directo al checkout. Así el test es más rápido y más enfocado en lo que importa.

5. Evitá lógica condicional dentro de los tests

TS
// MAL: un test no deberia "decidir" que camino tomar
cy.get("body").then(($body) => {
  if ($body.find('[data-cy="banner"]').length > 0) {
    cy.get('[data-cy="banner"]').click();
  }
});
// Si esto pasa seguido, el test no es deterministico: indica un problema
// en la app (o en como se prepara el estado) que conviene resolver antes.

6. Mantené los tests cortos y enfocados en un solo comportamiento

En vez de...Mejor...

14 Errores comunes a evitar

Estos son los errores que más se repiten en equipos que recién empiezan con Cypress (¡y a veces hasta en equipos con experiencia!). Tenerlos identificados ahorra horas de frustración.

ERROR COMÚN Encadenar comandos cy.get() muy largos y frágiles

Selectores como cy.get("div > div:nth-child(3) > span.value") se rompen con cualquier cambio mínimo de maquetado. Solución: pedí (o agregá vos mismo) atributos data-cy en los componentes relevantes.

ERROR COMÚN Asumir que .then() es asíncrono como una Promise normal

Es fácil pensar que podés mezclar libremente async/await con comandos de Cypress. No funciona así: dentro de un test, dejá que Cypress maneje el orden con su cola de comandos y .then(), no mezcles paradigmas.

ERROR COMÚN Reusar estado del navegador entre tests sin darte cuenta

Desde Cypress 12+, las cookies, localStorage y sessionStorage se limpian automáticamente entre tests por defecto. Si tu proyecto usa una versión vieja, o desactivaste ese comportamiento, un test puede "heredar" sesión de otro y dar falsos positivos.

ERROR COMÚN Usar selectores XPath cuando no hace falta

Cypress no trae XPath por defecto (existe un plugin), y en la gran mayoría de los casos CSS alcanza y sobra. Reservá XPath sólo para casos realmente imposibles de resolver con CSS (muy poco frecuentes).

ERROR COMÚN Tipar todo como any para que TypeScript deje de quejarse

Pierde el sentido de usar TypeScript. Si una fixture o una respuesta de API no tiene tipo claro, definí la interface correspondiente: te va a ahorrar bugs tontos (typos de propiedades) todo el tiempo.

ERROR COMÚN No limpiar/resetear el estado del backend entre corridas

Si tus tests crean datos (usuarios, pedidos) y no los limpian, las corridas sucesivas empiezan a fallar por datos duplicados o inconsistentes. Usá tareas de seed/reset vía cy.task() o endpoints de test dedicados.

ERROR COMÚN Ignorar los videos y screenshots que genera Cypress en CI

Cuando un test falla en CI, Cypress guarda automáticamente una captura de pantalla (y video, si está habilitado). Es la forma más rápida de entender qué pasó sin tener que reproducir el fallo localmente.

Fundamentos

1 ¿Qué es Playwright?

Playwright es un framework de testing E2E de código abierto creado por Microsoft. Permite controlar Chromium, Firefox y WebKit (el motor de Safari) con una única API, usando JavaScript/TypeScript, Python, Java o .NET. A diferencia de Cypress, Playwright controla el navegador desde un proceso Node.js externo mediante un protocolo propio (basado en WebSocket/CDP), lo que le da soporte nativo a múltiples navegadores, múltiples pestañas y múltiples dominios sin restricciones.

¿Por qué lo creó Microsoft?

El equipo que construyó Playwright es en parte el mismo que trabajó en Puppeteer en Google. El objetivo era resolver tres limitaciones concretas que tenían con las herramientas existentes:

  • Multi-navegador de verdad: Puppeteer solo controla Chromium. Playwright controla los tres motores principales desde la misma API.
  • Aislamiento automático: cada test corre en un BrowserContext nuevo, sin necesidad de limpiar cookies ni storage manualmente.
  • async/await nativo: el modelo de comandos es JavaScript puro; no hay cola de comandos interna que el programador no vea.

2 Diferencias clave para quien viene de Cypress

Si ya leíste la pista de Cypress (o venís de usarlo), esta sección es la más importante. Playwright no es Cypress con el nombre cambiado: tiene un modelo mental distinto en varios puntos clave. Ignorar esto genera errores silenciosos difíciles de debuggear.

1. Asincronía: el cambio más importante

En Cypress los comandos forman una cola interna; Cypress los ejecuta en orden y nunca escribís await en un test normal. En Playwright todo es async/await de verdad: cada interacción retorna una Promise real.

CypressPlaywright
ModeloCola interna de comandosPromesas nativas de Node.js
¿Usás await?Casi nunca (y si lo ponés, puede romper)Siempre en cada acción
Error silencioso típicoUsar .then() en el lugar equivocadoOlvidar el await
Por qué importaEl orden lo maneja Cypress, no vosVos controlás el flujo explícitamente
TS
// CYPRESS — sin await, cola interna
it('loguea', () => {
  cy.visit('/login');
  cy.get('[data-cy=email]').type('ana@mail.com');
  cy.get('[data-cy=pass]').type('clave123');
  cy.get('[data-cy=submit]').click();
  cy.url().should('include', '/dashboard');
});

// PLAYWRIGHT — async/await, promesas reales
test('loguea', async ({ page }) => {
  await page.goto('/login');
  await page.getByLabel('Email').fill('ana@mail.com');
  await page.getByLabel('Contraseña').fill('clave123');
  await page.getByRole('button', { name: 'Ingresar' }).click();
  await expect(page).toHaveURL(/dashboard/);
});
ATENCIÓN El error más común: olvidar el await

Si escribís page.click() sin await, el test no falla inmediatamente: la Promise queda flotando, el código sigue ejecutándose, y obtenés comportamientos raros difíciles de entender. La sección 14 cubre este error en detalle con ejemplos de lo que pasa. Por ahora, la regla simple: todo lo que empieza con page., locator. o expect() lleva await.

2. El sistema de test runner propio

TS
// Cypress usa Mocha (describe / it)
describe('Suite', () => {
  before(() => { /* ... */ });
  beforeEach(() => { /* ... */ });
  it('caso 1', () => { /* ... */ });
  it('caso 2', () => { /* ... */ });
});

// Playwright tiene su propio runner (test.describe / test)
import {{ test, expect }} from '@playwright/test';

test.describe('Suite', () => {
  test.beforeAll(async () => { /* ... */ });
  test.beforeEach(async ({ page }) => { await page.goto('/'); });
  test('caso 1', async ({ page }) => { /* ... */ });
  test('caso 2', async ({ page }) => { /* ... */ });
});

3. BrowserContext: el aislamiento que Cypress no tiene por defecto

En Cypress, si un test escribe en localStorage, ese estado puede filtrarse al siguiente test (aunque Cypress 12+ mejoró esto). En Playwright, cada test recibe un BrowserContext nuevo y destruido al terminar: aislamiento garantizado sin configurar nada.

4. Locators vs. cy.get()

Cypress encuentra elementos con selectores CSS via cy.get(). Playwright tiene su propia capa de Locators: métodos orientados a cómo el usuario percibe la interfaz, no a cómo está construida en HTML.

TS
// Cypress: selector CSS directo
cy.get('[data-cy="submit"]').click();

// Playwright: locator orientado al usuario (preferido)
await page.getByRole('button', { name: 'Enviar' }).click();

// Playwright: si necesitás CSS igual (válido, pero no es el primer recurso)
await page.locator('[data-testid="submit"]').click();
TIP ¿Por qué locators orientados al usuario?

Un test que busca por rol y nombre accesible falla si el botón desaparece o cambia su texto visible — que es exactamente lo que querés detectar. Un test que busca por clase CSS puede pasar aunque el botón sea completamente inaccesible para un lector de pantalla. Los locators de Playwright te hacen testear y mejorar la accesibilidad al mismo tiempo.

5 Locators: la forma Playwright de encontrar elementos

Los Locators son la forma de Playwright de encontrar elementos. A diferencia de Cypress (que usa selectores CSS vía cy.get()), Playwright propone primero los locators orientados a cómo el usuario percibe la interfaz, y deja CSS/XPath como último recurso. Esta jerarquía tiene una razón: los locators semánticos hacen tests más robustos y mejoran la accesibilidad del código.

1. getByRole — el rey de los locators

Busca por rol ARIA y nombre accesible. Refleja exactamente lo que ve un lector de pantalla y lo que espera un usuario.

Por defecto getByTestId() busca data-testid. Podés cambiarlo en playwright.config.ts: use: {{ testIdAttribute: 'data-cy' }}. Así usás getByTestId() con los atributos data-cy que ya tenés en tu HTML.

6 Anatomía de un test

Un test de Playwright es una función async que recibe fixtures inyectadas (como page, context, request) y usa await para cada interacción. El runner de Playwright crea y destruye automáticamente el BrowserContext de cada test.

Estructura básica

TS
import { test, expect } from '@playwright/test';

// test.describe agrupa tests relacionados (equivalente a describe() en Cypress/Mocha)
test.describe('Carrito de compras', () => {

  // beforeAll: corre una sola vez antes de todos los tests del bloque
  test.beforeAll(async ({ browser }) => {
    // ej: crear datos en una BD de test
  });

  // beforeEach: corre antes de CADA test (fixture page es nueva por test)
  test.beforeEach(async ({ page }) => {
    await page.goto('/productos');
  });

  // El test en sí: función async que recibe fixtures entre {}
  test('agrega un producto al carrito', async ({ page }) => {
    await page.getByRole('button', { name: 'Agregar' }).first().click();
    await expect(page.getByTestId('cart-count')).toHaveText('1');
  });

  test('muestra precio total actualizado', async ({ page }) => {
    await page.getByRole('button', { name: 'Agregar' }).first().click();
    await expect(page.getByTestId('cart-total')).toContainText('$');
  });

  test.afterEach(async ({ page }) => {
    // Playwright destruye el BrowserContext aqui de todas formas,
    // pero podés hacer cleanup adicional si querés
  });
});

La fixture page — qué es y de dónde viene

En Playwright, page no es un global: es una fixture inyectada por el runner. Cada test recibe su propia instancia de Page, creada en un BrowserContext nuevo y destruida al terminar. Nunca necesitás importar ni inicializar page: simplemente la declarás entre los parámetros de la función de test.

TS
// page, context, browser, request — fixtures built-in disponibles en cualquier test
test('ejemplo con múltiples fixtures', async ({ page, context, request }) => {
  // page: la Page activa (tab del navegador)
  await page.goto('/login');

  // context: el BrowserContext (útil para crear pages adicionales)
  const otraPage = await context.newPage();
  await otraPage.goto('/admin');

  // request: para llamadas HTTP puras sin navegador
  const resp = await request.get('/api/health');
  expect(resp.ok()).toBeTruthy();
});

test.only, test.skip y test.fixme

TS
// Correr solo este test (útil mientras debuggeás)
test.only('este es el único que corre', async ({ page }) => { /* ... */ });

// Saltar este test
test.skip('pendiente de implementar', async ({ page }) => { /* ... */ });

// Marcar como roto (corre y se espera que falle)
test.fixme('este test falla conocido', async ({ page }) => { /* ... */ });

test.step — para legibilidad en tests largos

TS
test('checkout completo', async ({ page }) => {
  await test.step('Agregar producto al carrito', async () => {
    await page.goto('/productos');
    await page.getByRole('button', { name: 'Agregar' }).first().click();
  });

  await test.step('Ir al checkout', async () => {
    await page.getByRole('link', { name: 'Ver carrito' }).click();
    await page.getByRole('button', { name: 'Pagar ahora' }).click();
  });

  await test.step('Verificar confirmación', async () => {
    await expect(page.getByRole('heading', { name: 'Pedido confirmado' })).toBeVisible();
  });
});
TIP test.step aparece en el trace y el reporte

Los pasos nombrados con test.step aparecen como secciones diferenciadas en el Trace Viewer y el reporte HTML. Es la forma más directa de hacer un test largo legible tanto para quien lo escribe como para quien lo lee en el reporte.

8 Aserciones web-first

Las aserciones de Playwright se llaman web-first assertions porque reintentan automáticamente hasta que la condición se cumple o vence el timeout configurado. Es el equivalente al .should() de Cypress, pero con sintaxis estándar de Jest/expect y soporte de async/await.

ATENCIÓN Siempre expect(locator), nunca expect(await locator.textContent())

9 Datos de prueba y fixtures de Playwright

Playwright tiene dos conceptos separados que se llaman "fixture" y es importante no confundirlos:

  • Fixtures de datos: archivos JSON (o cualquier formato) con datos de prueba estáticos. Concepto análogo al de Cypress.
  • Fixtures de Playwright: un sistema de inyección de dependencias basado en test.extend() para compartir setup entre tests. Este es el concepto poderoso y único de Playwright, sin equivalente en Cypress.

1. Fixtures de datos — archivos JSON

La forma más simple: archivos de datos que importás directamente en los tests.

tests/fixtures/usuarios.json
{
  "usuarios": [
    { "email": "ana@mail.com",   "password": "clave123", "rol": "admin" },
    { "email": "beto@mail.com",  "password": "pass456",  "rol": "viewer" }
  ]
}
tests/login.spec.ts
// Tipar los datos con TypeScript
interface Usuario {
  email:    string;
  password: string;
  rol:      'admin' | 'viewer';
}
interface UsuariosFixture {
  usuarios: Usuario[];
}

// Importar y usar en el test
import usuariosData from '../fixtures/usuarios.json';
const data = usuariosData as UsuariosFixture;

test('admin puede acceder al panel', async ({ page }) => {
  const admin = data.usuarios.find(u => u.rol === 'admin')!;
  await page.goto('/login');
  await page.getByLabel('Email').fill(admin.email);
  await page.getByLabel('Contraseña').fill(admin.password);
  await page.getByRole('button', { name: 'Ingresar' }).click();
  await expect(page).toHaveURL(/admin/);
});

2. Fixtures de Playwright — inyección de dependencias

Este es el sistema propio de Playwright. Funciona con test.extend(): definís "fixtures" como funciones que reciben otras fixtures y las proporcionan al test. El runner se encarga de crear, inyectar y destruir cada fixture en el orden correcto.

INFO ¿Por qué esto no existe en Cypress?

Cypress resuelve el setup compartido con beforeEach() y custom commands. Playwright eligió un modelo más estructurado: las fixtures son explícitas (aparecen en la firma del test), componibles (una fixture puede usar otras) y con teardown incorporado (el cleanup vive en la misma función, no en afterEach).

tests/fixtures/auth.ts
// tests/fixtures/auth.ts
import { test as base, Page } from '@playwright/test';

// Tipo de tus fixtures custom
interface MisFixtures {
  paginaLogueada: Page;
  usuarioAdmin:   { email: string; nombre: string };
}

// Extender el test base con tus fixtures
export const test = base.extend<MisFixtures>({

  // Fixture que autentica al usuario antes de cada test
  paginaLogueada: async ({ page }, use) => {
    // --- SETUP ---
    await page.goto('/login');
    await page.getByLabel('Email').fill('ana@mail.com');
    await page.getByLabel('Contraseña').fill('clave123');
    await page.getByRole('button', { name: 'Ingresar' }).click();
    await page.waitForURL(/dashboard/);

    // "use" hace disponible la fixture al test
    await use(page);

    // --- TEARDOWN (opcional, el BrowserContext se destruye de todas formas) ---
    await page.goto('/logout');
  },

  // Fixture de datos: no necesita teardown
  usuarioAdmin: async ({}, use) => {
    await use({ email: 'ana@mail.com', nombre: 'Ana García' });
  },
});

// Re-exportar expect para que los tests solo importen desde acá
export { expect } from '@playwright/test';
tests/dashboard.spec.ts
// tests/dashboard.spec.ts — importa el test extendido, no el original
import { test, expect } from './fixtures/auth';

// paginaLogueada está disponible automáticamente, sin beforeEach
test('dashboard muestra el nombre del usuario', async ({ paginaLogueada, usuarioAdmin }) => {
  await expect(paginaLogueada.getByText(usuarioAdmin.nombre)).toBeVisible();
});

test('puede acceder a configuración', async ({ paginaLogueada }) => {
  await paginaLogueada.getByRole('link', { name: 'Configuración' }).click();
  await expect(paginaLogueada).toHaveURL(/configuracion/);
});
BUENA PRÁCTICA Composición de fixtures: la ventaja real

Una fixture puede recibir otras fixtures. Podés tener paginaAdmin que use paginaLogueada, que use page. El runner resuelve el grafo de dependencias automáticamente. El resultado: tests muy legibles y setup completamente reutilizable sin globals ni beforeEach anidados.

10 Page Object Model y fixtures custom

En Cypress, los custom commands (Cypress.Commands.add()) son el mecanismo para reutilizar lógica de UI. En Playwright, ese rol lo cumple el Page Object Model (POM) combinado con fixtures. No existe el concepto de "commands globales": la reutilización es explícita y tipada.

Un Page Object básico

pages/LoginPage.ts
// pages/LoginPage.ts
import { Page, Locator } from '@playwright/test';

export class LoginPage {
  // Locators como propiedades de clase: se definen una vez, se usan en muchos tests
  readonly emailInput:  Locator;
  readonly passInput:   Locator;
  readonly submitBtn:   Locator;
  readonly errorMsg:    Locator;

  constructor(private page: Page) {
    this.emailInput = page.getByLabel('Email');
    this.passInput  = page.getByLabel('Contraseña');
    this.submitBtn  = page.getByRole('button', { name: 'Ingresar' });
    this.errorMsg   = page.getByTestId('login-error');
  }

  async goto() {
    await this.page.goto('/login');
  }

  async login(email: string, password: string) {
    await this.emailInput.fill(email);
    await this.passInput.fill(password);
    await this.submitBtn.click();
  }

  async loginAndWait(email: string, password: string) {
    await this.login(email, password);
    await this.page.waitForURL(/dashboard/);
  }
}

11 Interceptación y mock de red

Playwright intercepta y modifica requests de red con page.route(), el equivalente de cy.intercept() en Cypress. La API es async/await, y tiene soporte para modificar headers, cuerpo, status code, o directamente abortar la request.

Mockear una respuesta completa

TS
test('muestra la lista mockeada de usuarios', async ({ page }) => {
  // Registrar el handler ANTES de navegar
  await page.route('/api/usuarios', async route => {
    await route.fulfill({
      status: 200,
      contentType: 'application/json',
      body: JSON.stringify([
        { id: 1, nombre: 'Ana García',  email: 'ana@mail.com' },
        { id: 2, nombre: 'Beto López',  email: 'beto@mail.com' },
      ]),
    });
  });

  await page.goto('/usuarios');
  await expect(page.getByRole('listitem')).toHaveCount(2);
  await expect(page.getByText('Ana García')).toBeVisible();
});

Modificar (no reemplazar) la respuesta real

TS
test('inyecta un campo extra en la respuesta real', async ({ page }) => {
  await page.route('/api/perfil', async route => {
    // Obtener la respuesta real del servidor
    const response = await route.fetch();
    const json = await response.json();

    // Modificar y reenviar
    json.badge = 'beta-tester';
    await route.fulfill({ json });
  });

  await page.goto('/perfil');
  await expect(page.getByTestId('badge')).toHaveText('beta-tester');
});

Simular errores de red

TS
// Abortar la request (simula corte de red)
await page.route('/api/usuarios', route => route.abort());

// Responder con un error HTTP
await page.route('/api/usuarios', route =>

);

// Pasar la request al servidor sin modificar (útil para desactivar parcialmente)

Esperar una request o respuesta

TS
// Patrón: lanzar la acción y esperar la response en paralelo

// Forma simple: solo esperar la URL

Multi-contexto en un solo test: simular dos usuarios

Un caso común en aplicaciones colaborativas: verificar que la acción de un usuario se refleja para otro usuario en tiempo real.

TS
test('dos usuarios ven el mismo documento compartido', async ({ browser }) => {
  // Crear dos contextos completamente aislados
  const ctxUsuario1 = await browser.newContext();
  const ctxUsuario2 = await browser.newContext();

  const pageUsuario1 = await ctxUsuario1.newPage();
  const pageUsuario2 = await ctxUsuario2.newPage();

  // Usuario 1 se loguea y crea un documento
  await pageUsuario1.goto('/login');
  await pageUsuario1.getByLabel('Email').fill('ana@mail.com');
  await pageUsuario1.getByLabel('Contraseña').fill('clave123');
  await pageUsuario1.getByRole('button', { name: 'Ingresar' }).click();
  await pageUsuario1.getByRole('button', { name: 'Nuevo documento' }).click();

  // Obtener el link para compartir
  const shareUrl = await pageUsuario1.getByTestId('share-url').inputValue();

  // Usuario 2 accede al documento con el link
  await pageUsuario2.goto(shareUrl);
  await expect(pageUsuario2.getByRole('heading', { name: 'Documento sin título' })).toBeVisible();

  // Usuario 1 edita, usuario 2 ve el cambio
  await pageUsuario1.getByRole('textbox', { name: 'Título' }).fill('Mi documento');
  await expect(pageUsuario2.getByRole('heading', { name: 'Mi documento' })).toBeVisible();

  // Cleanup
  await ctxUsuario1.close();
  await ctxUsuario2.close();
});

13 Buenas prácticas

Las buenas prácticas de Playwright comparten el espíritu de las de Cypress (independencia entre tests, sin sleeps manuales, datos de prueba robustos), pero las herramientas concretas son distintas.

1. Usá locators semánticos antes que CSS o XPath

TS
// MAL: CSS frágil — si el equipo de diseño cambia la clase, el test se rompe
await page.locator('.btn-primary.submit').click();

// MAL: XPath — imposible de leer y mantener
await page.locator('xpath=//div[@class="form"]//button[2]').click();

// BIEN: rol y nombre accesible
await page.getByRole('button', { name: 'Enviar formulario' }).click();

// BIEN si no hay locator semántico: data-testid explícito
await page.getByTestId('submit-button').click();

2. Usá aserciones web-first, no aserciones sobre valores obtenidos

TS
// MAL: sin auto-retry — puede fallar por timing aunque el elemento esté a punto de aparecer
const text = await page.getByTestId('resultado').textContent();
expect(text).toBe('Completado');

// BIEN: auto-retry — espera hasta que se cumpla o venza el timeout
await expect(page.getByTestId('resultado')).toHaveText('Completado');

3. Nunca uses waitForTimeout

TS
// MAL: espera ciega, lenta y frágil
await page.waitForTimeout(3000);
await expect(page.getByTestId('resultado')).toBeVisible();

// BIEN: esperá la condición real
await expect(page.getByTestId('resultado')).toBeVisible(); // auto-retry incluido

// BIEN si esperás una request:
await Promise.all([
  page.waitForResponse('/api/calcular'),
  page.getByRole('button', { name: 'Calcular' }).click(),
]);
await expect(page.getByTestId('resultado')).toBeVisible();

4. Usá test.step para tests largos

TS
// Sin steps: difícil de diagnosticar qué falló
test('flujo completo de compra', async ({ page }) => {
  await page.goto('/');
  await page.getByRole('link', { name: 'Productos' }).click();
  // ... 30 líneas más sin estructura visible

  // CON steps: el reporte HTML y el Trace Viewer los muestran separados
  await test.step('Agregar al carrito', async () => {
    await page.getByRole('button', { name: 'Agregar' }).first().click();
    await expect(page.getByTestId('cart-count')).toHaveText('1');
  });
  await test.step('Checkout', async () => {
    await page.getByRole('link', { name: 'Carrito' }).click();
    await page.getByRole('button', { name: 'Pagar' }).click();
  });
});

5. Aprovechá el aislamiento automático de BrowserContext

TS
// En Cypress había que limpiar estado entre tests:
// beforeEach(() => { cy.clearCookies(); cy.clearLocalStorage(); });

// En Playwright NO hace falta: cada test recibe un contexto nuevo
test('test A', async ({ page }) => {
  // Este test escribe en localStorage
  await page.evaluate(() => localStorage.setItem('tema', 'oscuro'));
});

test('test B', async ({ page }) => {
  // localStorage está limpio: el contexto es nuevo
  const tema = await page.evaluate(() => localStorage.getItem('tema'));
  expect(tema).toBeNull(); // pasa siempre
});

6. Usá data-testid cuando no hay locator semántico

CSS
<!-- En el HTML: cuando no hay rol ni label natural -->
<div data-testid="cart-count" aria-label="Ítems en el carrito">3</div>
TS
// En el test:
await expect(page.getByTestId('cart-count')).toHaveText('3');

// Configurar un atributo alternativo en playwright.config.ts:
// use: { testIdAttribute: 'data-cy' }  → getByTestId() busca data-cy=
BUENA PRÁCTICA Jerarquía de decisión para elegir un locator

1° getByRole + nombre accesible — 2° getByLabel (formularios) — 3° getByText / getByPlaceholder — 4° getByTestId — 5° page.locator(css) — Último recurso: XPath.

14 Errores comunes a evitar

1. Olvidar el await — el error más común

Este es, con diferencia, el error más frecuente para quien empieza con Playwright, especialmente si viene de Cypress donde nunca se usa await.

TS
// MAL: la Promise queda flotando — el click no se espera
test('click sin await', async ({ page }) => {
  await page.goto('/');
  page.getByRole('button', { name: 'Enviar' }).click(); // ← falta await
  // El test puede pasar o fallar de forma no determinística
  await expect(page).toHaveURL(/confirmacion/);
});

// BIEN:
test('click con await', async ({ page }) => {
  await page.goto('/');
  await page.getByRole('button', { name: 'Enviar' }).click(); // ← await
  await expect(page).toHaveURL(/confirmacion/);
});
TIP Activá la regla ESLint para detectarlo automáticamente

El plugin eslint-plugin-playwright tiene la regla playwright/no-floating-promises que detecta estos casos. También @typescript-eslint/no-floating-promises de la config base de TypeScript funciona. Configurá el linter desde el inicio.

2. No usar el Trace Viewer ni el UI Mode para debuggear

BASH

npx playwright test --debug

npx playwright show-report

# Opción 4: Forzar generación de trace
npx playwright test --trace on

3. Usar waitForTimeout en vez de esperar la condición real

TS
// MAL: si la animación tarda más de 500ms en otro entorno, el test falla
await page.waitForTimeout(500);
await expect(page.getByTestId('spinner')).not.toBeVisible();

// BIEN: la aserción web-first reintenta sola hasta que se cumpla

4. No usar codegen para arrancar

BASH

5. Buscar con CSS cuando existe un locator semántico

TS
// MAL: depende de la estructura del HTML
await page.locator('form.login-form > div:nth-child(2) > input').fill('ana@mail.com');

// BIEN: independiente de la estructura HTML
await page.getByLabel('Email').fill('ana@mail.com');

6. No usar storageState para reutilizar autenticación

TS
// MAL: hacer login en CADA test cuando ya se probó que funciona
test.beforeEach(async ({ page }) => {
  await page.goto('/login');

  await page.getByLabel('Contraseña').fill('clave123');

});

// BIEN: login una sola vez, guardar estado, reutilizarlo
// (ver sección 12 para el patrón completo con auth.setup.ts)

7. Confundir fixtures de datos con fixtures de Playwright

INFO Son dos conceptos distintos

"Fixture" en Playwright puede significar: (a) un archivo de datos JSON con datos de prueba, o (b) el sistema de inyección de dependencias de test.extend(). Playwright fue diseñado para correr en CI desde el día uno: los tests corren headless por defecto, la paralelización es nativa (sin necesitar un servicio externo), y el reporter HTML incluye traces, videos y capturas de pantalla de los fallos.
